# WorkerOS — Decisions Log

Architecture Decision Records. Each entry: the fork, the choice, why, and what it rules out. Deviations in implementation must be logged here against the original rationale.

Status legend: **[locked]** settled for v1 · **[deferred]** intentionally out of v1 · **[revisit]** may change with new information.

---

### ADR-001 — Execution engine: host engine, not a shipped engine · [locked]
**Choice:** Guest code runs on the host worker's own JS engine (V8/SpiderMonkey/JSC), not a JS engine we ship (Boa/QuickJS) in v1.
**Why:** Lighter (no engine bundle), faster, and fully spec-complete JS for free. Matches the WebContainer/NodePod model. A pure-Rust engine (Boa) can slot in later behind the same execution trait for stronger isolation.
**Rules out:** Guaranteed capability isolation by default (host engine shares the worker realm). Mitigated by per-worker processes + the `Membrane` level.

### ADR-002 — es-runtime / V8 isolates are the *server* tier, not the browser engine · [locked]
**Choice:** V8 isolates (and thus `es-runtime`) are excluded from the in-browser path; reserved for a separate native/server tier.
**Why:** V8 does not compile to `wasm32`; a web page cannot create child isolates. Isolates only exist to a native embedder. They remain ideal for the adversarial-isolation / full-Node-fidelity server product, which is deliberately a different thing.
**Rules out:** True (native) sandboxing and Spectre resistance in the browser product.

### ADR-003 — Topology: kernel worker + one worker per process · [locked]
**Choice:** Kernel boots in a single worker; every spawned program gets its own worker. (Not a single cooperative worker, even for MVP.)
**Why:** Makes `kill` real (`worker.terminate()` is the only way to stop a runaway synchronous loop from outside the realm), gives true parallelism, and provides per-process memory isolation from day one. Cost — heavier spawn, `postMessage`/SAB comms — is acceptable and buys an honest process model.
**Rules out:** Ultra-cheap in-realm "processes." Requires cross-origin isolation for SAB (ADR-010).

### ADR-004 — Rust kernel is the single authority · [locked]
**Choice:** Rust owns VFS, process table, module resolution, scheduling, and capability granting. The host JS shim only executes handed-over source and reports back.
**Why:** One authoritative core (consistent with OpenTF DBMS/es-runtime instincts). Prevents logic leaking host-side, keeps the execution strategy swappable, and keeps the kernel testable independent of the browser.
**Rules out:** JS-side module resolution / capability decisions.

### ADR-005 — ABI: WASI Preview 1 floor + `otf:*` ceiling · [locked]
**Choice:** Use WASI P1 for everything it can express; add custom `otf:*` calls *only* where WASI has no primitive.
**Why:** WASI-only cannot express process creation, kill, or IPC — but WorkerOS is multi-process by definition (ADR-003), so it necessarily exceeds WASI P1 the moment there is more than one process. WASI is the *userland* ABI; `otf:*` is the *kernel* ABI WASI never specified. WASI-first buys the entire `wasm32-wasi` ecosystem unmodified; the extension surface stays minimal and principled.
**Rejected alternatives:**
- *WASI-only:* impossible past a single process.
- *WASI Preview 2 only:* younger tooling, heavier (typed WIT/component host), and still lacks browser-specific primitives (SW networking, worker-spawn). Not worth it for v1.
- *Fully custom ABI (`wasm32-otf`):* elegant but strands you in an ecosystem of one — nothing compiles to it (the WasmLinux adoption trap).
**Rules out:** nothing needed; forward path to Preview 2 / components stays open.

### ADR-006 — `otf:*` v1 surface is exactly three calls · [locked]
**Choice:** v1 kernel ABI = `otf:spawn`, `otf:kill`, `otf:ipc_open`. Networking/preview/GPU are separate deferred namespaces.
**Why:** These three are the unavoidable minimum forced by a multi-process OS. Keeping v1 to three prevents ABI bloat and keeps the WASI-compat surface pure.
**Deferred:** `otf:net`, `otf:preview`, `otf:gpu`, live-reload.

### ADR-007 — Node is a guest tenant, kernel is Node-agnostic · [locked]
**Choice:** The kernel knows processes/files/syscalls only. All Node semantics (`require`, `node_modules` resolution, `fs`/`http`/`path` shims) live in the guest-side node layer (`workeros-programs/node`). `npm` is likewise a guest program (`workeros-programs`, installed at `/bin/npm`), not a kernel feature.
**Why:** This is *the* differentiator from NodePod (a Node-runtime). If Node concepts leak into the kernel, WorkerOS becomes "NodePod-in-Rust" and loses its reason to exist. On real Linux, Node is just a program; same here.
**Rules out:** Kernel-level Node fast-paths.

### ADR-008 — Networking: Service-Worker simulation, no raw sockets · [locked]
**Choice:** No `socket`/`listen`. "Servers" register routes; a same-origin Service Worker intercepts `fetch` to the preview URL and routes into the kernel. Outbound `fetch` is CORS-bound; npm needs a CORS proxy. `net`/`dgram` unsupported.
**Why:** The browser forbids raw sockets; this is the platform ceiling, not a design preference. SW interception is the only faithful way to present a "server."
**Rules out:** Real TCP/UDP servers/clients, real DNS. Documented as simulation.

### ADR-009 — Isolation is a policy knob; not Spectre-proof · [locked]
**Choice:** One execution trait, selectable levels: `Full` (bare import), `Membrane` (SES/`ShadowRealm` + frozen intrinsics), `Wasm` (Boa, later). Per-worker isolation applies at all levels.
**Why:** Different callers have different threat models. Membrane gives capability denial without a separate engine; the worker gives preemption/OOM containment; a future WASM engine gives linear-memory containment.
**Rules out:** Defense against motivated side-channel (Spectre-class) attackers — that needs the native tier (ADR-002). Stated openly.

### ADR-010 — Cross-origin isolation required · [locked]
**Choice:** Depend on `SharedArrayBuffer` (+ `Atomics`) for the synchronous syscall path; require COOP/COEP headers from embedders.
**Why:** WASI guests expect blocking `fd_read`; a SAB ring buffer with `Atomics.wait` is the standard way to provide block-until-satisfied across workers.
**Rules out:** Deployment on origins that cannot set COOP/COEP. Documented as a platform requirement.

### ADR-011 — In-memory VFS first, persistence behind the same trait · [locked/deferred]
**Choice:** v1 VFS is in-memory (Rust inode tree behind a `Vfs` trait). IndexedDB persistence and COW overlays are later implementations of the same trait.
**Why:** Keeps MVP simple; the trait boundary means persistence adds no call-site churn.
**Deferred:** IndexedDB backend, snapshot/overlay.

### ADR-012 — Custom bash-flavored shell, not bash · [locked]
**Choice:** Implement `wsh`, a small shell supporting the subset real projects use. Not bash-compatible.
**Why:** Real bash is a native ELF binary needing CPU emulation — antithetical to lightweight. ~95% of `package.json` scripts and build invocations need only argv/pipes/redirects/`&&`.
**Rules out:** Complex `.sh` scripts (arrays, `[[ ]]`, param expansion, traps, functions). Documented.

### ADR-013 — Name & packaging · [locked]
**Choice:** Product **WorkerOS**. Crates/packages: `workeros-kernel`, `workeros-web`, `workeros-coreutils` (system binaries), `workeros-programs` (OS programs + the Node-compat runtime; absorbed the former `workeros-node` and `workeros-npm`). GitHub `opentf/workeros`. Apache-2.0.
**Why:** Name is descriptive (boots in a worker, is OS-shaped), on-brand with the `*OS` style, and verified free on npm (`workeros`, `@opentf/workeros`) and GitHub (`opentf/workeros`). "webOS" was rejected — LG trademark + existing OSS edition. The "Worker" halo (Cloudflare Workers) is adjacent but non-colliding; the tagline resolves it.
**Rules out:** —

### ADR-014 — License: Apache-2.0 · [locked]
**Choice:** Apache-2.0 for the core.
**Why:** It's embeddable infrastructure; permissive licensing maximizes adoption, consistent with the "open answer to proprietary WebContainer" positioning.
**Revisit:** A future higher-value layer could be dual-licensed, as with other OpenTF projects — but the core stays Apache-2.0.

### ADR-015 — Ring-buffer transport: Rust reference spec + JS SAB mirror · [revisit]
**Context (the fork):** The synchronous syscall path (ADR-010) is a `SharedArrayBuffer` ring buffer touched by two agents: the JS program worker (which must block in `Atomics.wait` on `fd_read`) and the Rust→wasm kernel worker. Two ways to own the framing: (a) run the ring-buffer index/protocol logic inside the wasm kernel over the SAB as wasm *shared* memory (requires the `atomics` target feature + shared-memory build + `memory.atomic.wait`), or (b) implement the framing as a small, dumb byte-mover on each JS worker end, with the Rust kernel owning only the *semantic* syscall handling.
**Choice (v1):** (b). The ring-buffer framing is specified and unit-tested in Rust (`workeros-kernel::ringbuf`) as the authoritative reference; the browser SAB transport (`workeros-web/src/ringbuffer.js`) is a thin mirror of that exact byte layout. Both ends are dumb byte movers; every *semantic* decision (what a syscall means, VFS/caps/resolution) stays in Rust.
**Why:** Keeps the kernel natively unit-testable now (`cargo test`, Phase 0 exit criterion) without a wasm-threads/shared-memory build; keeps INV-2 intact because the transport carries bytes, it does not make decisions. The two implementations share one wire spec (documented in `ringbuf.rs`) and are cross-checked (native concurrency test + browser round-trip).
**Cost / what it rules out (for now):** Two implementations of the *framing* to keep in sync — mitigated by the shared spec and tests. 
**Revisit when:** we adopt a wasm shared-memory kernel build; at that point option (a) can subsume the JS framing and this ADR should move to `[locked]` on option (a) or be retired.

### ADR-016 — Phase 2 JS tier uses the async control channel, not the SAB sync path · [revisit]
**Context (the fork):** The SAB ring buffer (ADR-010/-015) exists for *synchronous* syscalls — a WASI guest's `fd_read` must block until data arrives. Phase 2 runs **JS** guests, whose stdio does not block: `console.log` / `process.stdout.write` return `void`, and top-level `await` handles asynchrony. So how should a JS guest's syscalls reach the kernel — over the blocking SAB path, or over async `postMessage`?
**Choice:** JS-tier syscalls in Phase 2 travel over the async control channel (`postMessage`) program-worker → kernel-worker → main. The SAB sync path is reserved for the WASI/WASM tier (Phase 4), where blocking is a real requirement. The ring buffer stays built and tested (Phase 0) so it is ready when Phase 4 needs it.
**Why:** Using `Atomics.wait` for a JS guest that never blocks would add cost and complexity for no semantic gain. Routing through the kernel worker still preserves INV-2 (the wasm kernel makes every decision — cap checks, fd classification, VFS writes, exit codes); the program worker remains a dumb CPU. Topology note: the **kernel worker creates the program workers** (nested workers) and holds their handles, so `worker.terminate()` is available for a hard kill (INV-4/ADR-003).
**Rules out (for now):** A JS guest performing a *synchronous* blocking read (e.g. a synchronous `readFileSync` on a pipe that is not yet ready). Not needed by ordinary scripts in Phase 2; when a JS tenant needs it, it uses the same SAB path the WASI tier will.
**Revisit when:** Phase 4 wires the SAB path for WASI; reassess whether the JS tier should share it for uniformity.

### ADR-017 — Phase 3 coreutils are JS guest programs, not kernel builtins or WASI binaries · [revisit]
**Context (the fork):** Phase 3 needs `echo cat ls cp mv rm mkdir pwd env true false`. Three shapes were considered: (a) Rust builtins inside the kernel; (b) JS guest programs run as real processes; (c) Rust compiled to `wasm32-wasi` and run through the WASI host.
**Choice:** (b) — JS guest programs written against the native `sys` syscall ABI, each running in its own program worker as a real, `ps`-visible, killable process (INV-4). (Installed in `/sbin` as system binaries — see ADR-018.)
**Why:**
- (a) is wrong-shaped: a coreutil executing inside the kernel worker is not a separate process (breaks INV-4) and teaches the kernel the names `ls`/`cat`/… (breaks INV-1's small authoritative core). On real systems `/bin/ls` is userland, not the kernel.
- (c) is the right *long-term* form and the project's differentiator, but a real WASI guest's `fd_read` on a pipe must block synchronously — so it forces the WASM program-worker path **plus** the SAB synchronous-syscall path (ADR-010/-016) **plus** the WASI host to all land first. That is essentially all of Phase 4, pulled forward out of the gated order, just to run `echo`.
- The coreutils carry almost no logic: the substance (VFS, glob, path resolution, pipe semantics) is already Rust and native-tested; `echo`/`cat`/`ls` are ~15-line argv→syscall adapters. Writing them in Rust buys negligible test coverage and no correctness; the postMessage syscall hop, not the language, is the cost.
**Consequence:** "kernel Rust / userland JS" is the intended tier split, not an inconsistency. JS coreutils remain the lightweight default even after Phase 4; they coexist with dropped-in `wasm32-wasi` binaries (§5.1 tier table).
**Revisit when:** Phase 4 builds the WASI host — at which point the *marquee* demo is an **unmodified** off-the-shelf binary (`ripgrep`/`jq`), a far stronger proof of the WASI-first thesis than a hand-written `echo.rs`.

### ADR-018 — Coreutils are system binaries in `/sbin`, apart from `/bin` programs · [locked]
**Choice:** The coreutils install in **`/sbin`** (system binaries); OS/user programs like `npm` install in **`/bin`**. The command search path is `PATH=/bin:/sbin` (kernel `DEFAULT_PATH`), so bare names still resolve, and `/bin` wins so a user program can shadow a system one.
**Why:** Separating the OS internals from the general program namespace makes them read as untouchable and keeps `ls /bin` about installed programs, not plumbing. It mirrors the FHS intuition (system vs. general binaries) even though our split is by *ownership*, not by admin-vs-user.
**Future work (not yet enforced):** `/sbin` is currently only a *convention* — nothing physically prevents `rm /sbin/ls`. Real enforcement is a kernel-level **protected-prefix** check in the syscall layer (`unlink`/`rmdir`/`rename`/`write`/`open(create,truncate)` under a protected prefix → `EPERM`), plus a way to mark a VFS subtree read-only/immutable. Deferred; see PLAN Phase 7 (isolation/hardening).

### ADR-019 — One OS-programs package, not one package per program · [locked]
**Choice:** All installable `/bin` programs (`npm`, …) and the Node-compatible guest runtime live in a single package, **`@opentf/workeros-programs`** (with `./node` for the runtime), which absorbed the former `@opentf/workeros-npm` and `@opentf/workeros-node`. A small registry lists programs; the kernel worker installs the whole set at boot. Programs carry a `type` (`js` now, `wasm` later).
**Why:** Avoids a package-per-program sprawl; adding a program is one registry entry. `workeros-coreutils` stays separate as the *system* binaries (ADR-018).
**Future work:** A selectable install manifest (choose which builtins to include) — for now everything installs at boot.

### ADR-020 — Resource limits & fault isolation: the kernel accounts, the host enforces the clock · [locked]
**Context (the fork):** Nothing today bounds a guest. It can fork-bomb through `otf:spawn`, allocate until the tab OOMs, spin a synchronous loop forever (only `terminate()` can stop it), exhaust fds/pipes, or fill the VFS. Two headline use cases — **embedding in a SaaS product** and **running untrusted / AI-generated code** (§2) — make this a *safety* requirement, not a nicety. The design question is ownership: the limits fall into two families. **Accounting limits** (counts of processes, fds, bytes) are pure bookkeeping. **Temporal limits** (wall-clock CPU time, memory high-water) need a clock and the ability to `terminate()` a worker — and the wasm kernel has *neither* (no wall clock, no handle on the workers; the **kernel worker** on the JS side holds those, ADR-016).
**Choice:** Split by capability, mirroring ADR-015's dumb-byte-mover discipline — *Rust decides the numbers, the host supplies the mechanism it alone has*:
- **Kernel-authoritative (Rust, natively tested):** process-count caps (global + per-tree → `EAGAIN`, POSIX `fork` under `RLIMIT_NPROC`), open-fd caps (→ `EMFILE`), and VFS byte + inode quota (→ `ENOSPC`). A `ResourceLimits` policy object is granted at spawn alongside the `CapabilitySet` and **inherited by children** (a child's usage counts against its ancestors' budget); a `ResourceUsage` counter set is updated on every spawn / reap / fd-alloc / VFS write. All enforced at the seams that already exist — `Kernel::spawn` (the single process-creation choke point), the fd allocators, and the VFS write/create paths.
- **Host-enforced (kernel-worker JS watchdog):** wall-clock/idle time and memory high-water. The kernel worker samples each program worker (`performance.measureUserAgentSpecificMemory()`, available under the cross-origin isolation we already require per ADR-010; a liveness heartbeat for time) and on breach runs the **cooperative-then-hard** kill the `Ctrl-C` path already implements (deliver a cooperative signal, grace period, then `worker.terminate()`), records a **kill reason** on the process record, and reaps — closing pipe fds (EOF downstream) and restoring the TTY on the fault path exactly as an ordinary exit does today.
**Why:** The load-bearing DoS guards (fork-bomb, fd-bomb, disk-fill) are precisely the ones expressible as Rust bookkeeping, so they stay in the natively-testable kernel (INV-2, Phase-1 discipline) and cost almost nothing. Only the two limits that *genuinely* need a wall clock and preemption are pushed host-side, where the mechanism actually lives — and even there the host makes no *policy* decision: the kernel owns the numbers and the accounting; the host reports a breach back and asks the kernel to record the kill. Limits are the **quantitative** sibling of the `Membrane`'s **qualitative** capability denial (ADR-009); together they are the "safe sandbox" the AI-agent use case claims. Distinct kill reasons (OOM `137`, CPU-time, fault-crash vs. ordinary exit) give `ps`/`wait`/the shell an honest *why* — the observability half of the story.
**Honest surface (INV-5):** The memory ceiling is a **soft, sampled high-water mark**, not a hard allocator cap — a single synchronous huge allocation can still OOM the tab between samples. Stated openly; a hard cap arrives only with the future `Wasm`/Boa level (ADR-009). Time enforcement is **cooperative-first, not preemptive** time-slicing. Neither defends against an allocator deliberately racing the sampler (same Spectre-class honesty as ADR-009). Quotas are per-session until persistence (ADR-011) gives them a durable home.
**Rules out:** hard per-process memory sandboxing at the `Full`/`Membrane` levels; preemptive fair-share scheduling (the browser schedules workers — we cap *concurrency*, we do not time-slice).
**Revisit when:** the `Wasm` execution level (Boa) lands — a pure-Rust engine in linear memory can enforce a *hard* memory cap and cooperative fuel-based CPU metering inside the kernel, subsuming the host-side memory watchdog and moving that limit back across the ADR-015 line into Rust.
**Status (v1):** the kernel-authoritative accounting caps are **implemented and native-tested** (`workeros-kernel/limits.rs`) with recommended defaults — **128** live procs, **256** open fds/process, **256 MiB** + **100k** inodes of VFS storage — enforced in `Kernel::spawn` (`EAGAIN`), `ProcessCtx::alloc_fd` (`EMFILE`), and `MemVfs::write_at`/`alloc` (`ENOSPC`). The **host-side watchdog is implemented** (`kernel-worker.js` `watchdogTick`): CPU-time reaps with exit **152** (128+SIGXCPU), memory with **137**, each recording a **kill reason** in the kernel (`Kernel::mark_killed` → `ProcInfo.kill_reason`) and a shell-visible `Killed (…)` on the process's stderr; the escalation is the cooperative-then-hard two-phase (SIGTERM → grace → `terminate()`). Values are overridable per boot (`boot({ watchdog })` — the untrusted/AI-agent profile seam; `0` disables a limit); kernel caps remain hardcoded with `Kernel::boot_with_limits` as their override seam. Proven in `tools/watchdog.test.js` (spin reaped; responsive and syscall-blocked processes untouched; OOM path feature-gated).
**Deviation (implemented semantics of "CPU time"):** the budget is **continuous unresponsiveness**, not total wall-clock lifetime — the original 30s-wall-budget reading would kill every long-lived server. A process is *alive* if it makes syscalls, answers the kernel worker's liveness PING (its event loop turns), or is parked in a kernel-serviced blocking call (its sync-channel slot is non-idle — the kernel worker reads the SAB state directly, so a guest waiting on the kernel is never blamed for the kernel's wait). Only a worker showing none of these for `wall_time_ms` — a synchronous `for(;;)` — is reaped. Honest limitation: a guest blocked in its *own* `Atomics.wait` (user-level sync IPC, e.g. a worker-threads build-tool protocol) is indistinguishable from a spin; hosts running such workloads disable the budget (`wallTimeMs: 0`). The memory sampler is per-worker self-measurement (`performance.measureUserAgentSpecificMemory`), active only where the browser exposes it to workers — soft and sampled, exactly as this ADR already stated.

### ADR-021 — Networking: kernel moves bytes over port-keyed sockets; all HTTP is guest userland · [locked]
**Context (the fork):** ADR-008 settled the *ceiling* — no raw sockets; "servers" are a Service-Worker simulation — but not the *shape* of the subsystem that makes a dev server (Vite, and `http.createServer` generally) actually work end-to-end. The load-bearing question is **at what layer the kernel touches a connection.** Two families:
- **(a) Message-level bridge (WebContainer-lite):** the kernel/host carries a *structured* HTTP request in and expects a structured response out; the guest never sees wire bytes. Simplest, but it cannot carry a WebSocket — an `upgrade` hands the library a *raw socket* it frames itself, so HMR (`ws`) is impossible on this path — and it teaches the boundary about HTTP (against INV-1's spirit).
- **(b) Byte-stream sockets:** the kernel moves *opaque bytes* over a connection that is nothing but a pair of pipes keyed by a port; **all** HTTP/1.1 and WebSocket parsing/framing lives in the guest `node` layer. Faithful (real `net.Server`, `ws` upgrade, keep-alive all work), and the kernel stays not just Node-agnostic but *protocol*-agnostic.
**Choice:** **(b).** A "listening socket" is kernel state — a `port → (pid, listener)` registry in `workeros-kernel/net.rs`, sibling to the process table and `PipeTable` (INV-2: Rust owns authoritative state). The **data path reuses existing pipes** (§6.3): a connection is two `PipeId`s (`c2s` client→server, `s2c` server→client), so `fd_read`/`fd_write`/`fd_close` already stream it — **no new data-path syscall.** The kernel ABI grows by exactly the primitives a multi-process OS cannot otherwise express, mirroring ADR-006's discipline:
- `otf:net_listen(port) → listener` — claim a port; `EADDRINUSE` (a new errno) if held. Records `port → (pid, listener)`.
- `otf:net_accept(listener) → {rfd, wfd}` — the next inbound connection as a bound pipe pair. Event-driven for the JS tier, so it uses the **async control channel** (ADR-016), parking in a `pendingAccepts` list symmetric to `pendingReads`; the SAB blocking path (ADR-010) is reserved for a future WASI server. A new `OtfCall::NetListen` capability gates it.
- `otf:net_connect(port) → {rfd, wfd}` — a **loopback** connect *within the instance* (the SW-driven injector, and intra-OS clients). `ECONNREFUSED` if no listener. This is **not** outbound internet access — that stays browser `fetch`, CORS-bound, per ADR-008 (`http.request`/`get` map onto `fetch`, reusing curl's forbidden-header handling).
**The two worlds meet in one place (the injector).** The kernel worker exposes a host-callable "inject a connection to port N": it `net_connect`s a loopback pair, writes request bytes in, unblocks the parked `accept`, and reads response bytes out. The **Service Worker** (`preview-sw.js`, same-origin, extending today's `coi-serviceworker.js` role) intercepts `fetch` to the preview URL (`/__preview__/<port>/*`), serializes it to HTTP/1.1 request bytes, drives the injector, parses the response bytes, and returns a `Response`. All HTTP lives in the guest (`workeros-programs/node`): `net.Server`/`Socket` over the connection fds, `http.Server` (an HTTP/1.1 parser + `ServerResponse` serializer) on top, and `ws` **unmodified** because `upgrade` is a genuine byte stream.
**Honest surface (INV-5) — the one deviation:** a Service Worker **cannot** intercept `new WebSocket()`, so HMR cannot use a real browser→server WebSocket. The fix is a tiny client shim injected into served HTML that tunnels the HMR socket over `BroadcastChannel`/`postMessage` (the preview iframe is same-origin) to a `net_connect`+`upgrade` connection, so the guest `ws` server sees a normal upgrade. Documented openly as a *tunneled* transport, not a native WebSocket. An open listener/live connection also counts as pending work in the guest event loop (`event-loop.js`) so a server process does not report idle-exited.
**Reap on exit:** listeners and half-open connection pipes are freed through the existing process-reap seam (`mark_exited` → `close_all_io`), so a crashed server frees its port and its client sees EOF — nothing wedges or leaks (INV-6).
**Rules out:** real TCP/UDP servers/clients and real DNS (unchanged from ADR-008); a *native* browser WebSocket to an in-OS server (tunneled instead). **Native-testable (INV-2):** `listen`/`connect`/`accept` + the port registry + reap are pure kernel bookkeeping over `PipeTable`, proven in `cargo test` without a browser — the HTTP/preview/HMR layers are guest/host JS, tested in the headless browser.

### ADR-022 — Persistence: authoritative in-memory tree + write-behind projection with path-based durability · [locked]
**Context (the fork):** ADR-011 deferred persistence with a one-line plan — "an IndexedDB `Vfs` *implementation* behind the same trait." Building it exposed two problems that reshape the design. **(1)** The `Vfs` trait is **synchronous** (the syscall path and the SAB blocking channel, ADR-010, depend on immediate returns), but IndexedDB is **async and browser-only** — so an IndexedDB backend cannot sit *behind* the sync trait without making every read block on a DB round-trip, and it would drag browser I/O into the natively-tested kernel (against INV-2). **(2)** A whole-fs backend swap is all-or-nothing, but real usage wants a **hybrid**: source files should survive a reload while `/tmp` scratch and the huge, fully-reconstructible `node_modules` tree should *not* be serialized to storage on every change.
**Choice:** Persistence is a **projection of an authoritative in-memory tree**, not a backend swap; durability is a property of the **path**, not a per-inode/per-write flag; and the projection is a **content-addressed store in the ZFS/git shape**, not a whole-tree blob:
- **The in-memory `MemVfs` stays authoritative and synchronous** (unchanged hot path, still native-tested). Persistence is a background *write-behind*, and the kernel never touches IndexedDB — Rust decides *what* is durable and hands the host bytes-by-key; the host supplies the async storage mechanism it alone has (the same ADR-015/-020 "dumb byte mover" split).
- **Content-addressed projection.** File data is split into 64 KiB **chunks** keyed by their **SHA-256** (a dependency-free in-kernel `crate::hash`, `vfs.rs` `ChunkStore`), stored once and refcounted — identical chunks dedup across files *and* snapshots, and a write re-chunks so unchanged regions re-hash to the same keys (the persisted delta is just the changed chunks; the hash doubles as an integrity checksum, verified on load). A **manifest** (`b"WOM1"`) — the durable directory tree, inode metadata (mtime/ctime/btime), symlink targets, and each file's ordered chunk-hash list — is the root that ties the chunks together. The host stores chunks by hex hash (compressed with `deflate-raw`) plus the manifest; on each flush it writes only chunk hashes it doesn't already hold (delta). Boot loads the chunks (integrity-checked), then `hydrate_manifest` rebuilds the tree over them.
- **Snapshots + GC (ZFS-style).** A snapshot is a retained manifest that **increfs** every chunk it references, so a later working-tree edit or delete can't free bytes it still needs (copy-on-write by sharing, not copying). Named snapshots persist until destroyed; a rolling **auto-ring keeps the last 10** durable states (undo). Reclamation is **mark-sweep**: the kernel's `live_chunks` is the union of chunks referenced by the working tree **or** any retained snapshot, and the host deletes every stored chunk absent from it. The retained set is exported/imported (`b"WOSN"`) so snapshots survive a reload.
- **Durability is path-based (a `MountTable`, `vfs/mount.rs`)** — the Unix `tmpfs`-at-`/tmp` model. Longest-prefix match; default policy persists root `/` and marks `/tmp` (user scratch) and the boot-reinstalled OS trees (`/bin`, `/sbin`, `/lib`) ephemeral. A per-inode durability flag was **rejected**: it forces every writer to decide, needs rewriting on `rename` across a boundary, and the ino-based `write_at` seam has no path to check. Path-based resolution at *manifest* time makes cross-boundary `mv` correct for free and touches no inode state. An embedder can adjust the policy (`Kernel::mount(prefix, ephemeral)`).
- **When to persist:** a monotonic `generation` counter bumps on every content/structure mutation; the host write-behind re-persists only when it advances past the last-stored value (an idle OS does zero I/O), on a ~2s debounce plus a `visibilitychange`/`pagehide` flush. Auto-snapshots are taken on a coarser (~5 min) cadence.
**Honest surface (INV-5):** durability is **eventually-consistent** — up to ~2s (or the last un-flushed change on a hard crash) can be lost; the hide/unload flush trims but cannot guarantee it. Symlinks and mtime/ctime/btime **are** persisted (the VFS models them). Quotas remain per-session until a durable quota home lands. A snapshot *restore* replaces the persistent working tree wholesale (ephemeral subtrees untouched) and invalidates any open handles into it — restore is an offline-shaped operation.
**Native-testable (INV-2):** the `MountTable` (longest-prefix, carve-outs), the manifest round-trip (dedup + delta + metadata + symlinks), chunk refcount/COW, the snapshot lifecycle (retain-through-delete, restore, auto-ring eviction), the live-set/GC mark, and export/import are all pure kernel bookkeeping, proven in `cargo test`; the IndexedDB block store + compression + boot rehydration + reload-survival + GC sweep are proven in headless-browser tests (`tools/persistence.test.js`).
**Supersedes:** ADR-011's "IndexedDB `Vfs` behind the trait" phrasing; and this ADR's own earlier whole-tree-blob (`b"WOFS"` `snapshot`/`hydrate`) mechanism, now retired in favor of the content-addressed manifest+chunk store. The trait boundary still stands; the *persistence mechanism* is a projection over it.
**Revisit when:** the `Wasm`/Boa execution level (ADR-009) motivates a different chunk granularity, or a durable quota home / per-file overrides become necessary.

### ADR-023 — Pipes are bounded with parked writers, EPIPE/SIGPIPE, and streaming shell pipelines · [locked]
**Context (the fork):** The kernel pipe was an *unbounded* `VecDeque`: `write` always succeeded, whatever the reader was doing, and a pipe whose last reader had gone was never signalled. Two consequences. **(1)** No backpressure — `fast-producer | slow-consumer` grew kernel memory without limit, uncounted by any ADR-020 quota. **(2)** No termination — the classic Unix pipeline contract (`yes | head -1` ends because the writer gets `SIGPIPE`/`EPIPE` when `head` leaves) did not exist, so an early-exiting reader left the writer running forever. Separately, the shell's multi-stage pipelines didn't use kernel pipes at all: `runPipeline` was "collect and feed" (buffer stage *k*'s entire stdout in a JS array, then hand it to *k+1*), which is neither concurrent nor bounded, and can never terminate an infinite producer.
**Choice:** POSIX pipe semantics end to end, split along the ADR-020 ownership line (kernel decides, host supplies the mechanism):
- **Kernel (Rust, natively tested):** a pipe buffers at most **`PIPE_CAPACITY` = 64 KiB** (Linux's default). `write` accepts at most the free capacity and reports a **short count** (`WriteEffect::Pipe { nwritten }`, possibly 0) — never buffering beyond the cap. A write to a pipe whose last read end closed is **`EPIPE`** (`Errno::Pipe`, WASI value 64). Ends attach one spawn at a time (the shell wires the writer before the reader), so a never-yet-attached end reads as *would-block* — not EOF/EPIPE — and only an end that existed and departed is final (`had_reader`/`had_writer`).
- **Host (kernel worker):** a short pipe write **parks**, symmetric to the existing parked-read lists. A *blocking* write (SAB sync channel — now the default: the program worker routes `sys.write` through it, chunked) keeps the guest thread in `Atomics.wait` until every byte is accepted, so guests never see a short pipe write; a *fire-and-forget* write (pre-START fallback) queues its remainder per (pid, fd) in order. One **`pumpIo()`** loop replaces the scattered retry calls: any event that can advance a pipe (write, read, close, exit, injected connection) pumps all parked lists until stable.
- **SIGPIPE disposition:** on EPIPE the kernel worker applies the POSIX default — the writer is killed with **128+13** — unless the process registered a SIGPIPE handler (`sys.sighandle`), in which case the signal is delivered cooperatively and the write raises `EPIPE` (guest `node:fs` maps it; the WASI host returns errno 64). Reuses the exact caught-vs-default machinery the `Ctrl-C` path already had.
- **Shell:** a multi-stage pipeline whose stages are all plain external commands now runs **concurrently over real kernel pipes** (spawned writer-first; exit code is the last stage's). Stages that are builtins/functions/compounds — or whose command word needs expansion to resolve — keep the collect-and-feed fallback.
**Why:** Bounded pipes close the one DoS hole ADR-020's quotas missed, and EPIPE termination is a load-bearing behavior of every real shell — programs are *written* assuming it. Routing guest writes through the sync channel is what makes backpressure real (the producer's thread genuinely blocks) rather than cosmetic (an overflow queue that just moves the unbounded buffer host-side). The streaming pipeline makes the shell honest about the same contract; without it, kernel-level correctness was unreachable from the user-visible surface.
**Honest surface (INV-5):** collect-and-feed still applies when a builtin participates in a pipeline (`producer | while read …` buffers unboundedly in the driver) — acceptable because builtins are shell-internal, documented here. Partial-write atomicity below `PIPE_BUF` is not guaranteed for concurrent same-pipe writers (the shell never creates that shape today). A killed writer reports `141` (128+SIGPIPE), not Node's "unhandled `EPIPE` stream error, exit 1" — Node-fidelity refinement tracked in NODE_COMPAT.
**Rules out:** unbounded kernel/host buffering of pipe data anywhere on the write path; treating a short pipe write as complete.
**Native-testable (INV-2):** capacity clamp, short-count reporting, drain-then-resume, EPIPE-after-reader-departs, and the attach-order rules are pure kernel bookkeeping (`cargo test`); parking/pumping, the SIGPIPE disposition, streaming pipelines, and byte-integrity across park/drain cycles are proven in headless-browser tests (`tools/pipe-backpressure.test.js`).
**Revisit when:** job control / process groups land (SIGPIPE delivery should target the process group), or a WASI guest needs `poll_oneoff` over pipe readiness.

### ADR-024 — Outbound network is a kernel-granted capability, enforced at the worker boundary · [locked]
**Context (the fork):** The kernel's capability model (INV-2: "the kernel is the sole authority for capability granting") gated `otf:spawn/kill/ipc/net_listen` — but the most security-relevant capability of all, *talking to the internet*, never passed through it. Outbound HTTP is the browser's ambient `fetch` in the program worker (ADR-008: CORS-bound, how `npm install` reaches the registry), so every guest had unconditional egress: an "isolated" AI-agent process could exfiltrate anything it read. Three shapes were on the table:
- **(a) Kernel-mediated egress (`otf:net_fetch`):** every outbound request becomes a syscall the kernel can gate, audit, and proxy. Maximal, but it rewrites the guest `http`/`https`/npm stack, adds a copy across the worker boundary for every response body, and buys enforcement no stronger than (b) at the `Full` trust level — same-realm guest code sits beside the shim either way.
- **(b) A kernel-granted bit, host-enforced by stripping the globals:** the kernel owns a per-process `net_egress` capability; the program worker enforces a denial by deleting the egress surface (`fetch`, `XMLHttpRequest`, `WebSocket`, `EventSource`, `WebTransport`, and `Worker`/`SharedWorker`/`importScripts`, which could mint fresh globals) from the whole prototype chain and pinning non-configurable `undefined` shadows — *before any guest code evaluates*, so no reference can have been captured.
- **(c) Document it as host policy and move on:** honest, free, and useless to the sandbox story.
**Choice:** **(b)** now, with (a)'s syscall shape deferred to the `Membrane` level (ADR-009) where it becomes load-bearing. The kernel *decides* (INV-2): `CapabilitySet` gains `net_egress` (default **true** — the npm-install model is the ordinary case), `Kernel::spawn` echoes the granted bit in `Spawned`, and the worker is the dumb mechanism. **Capabilities now inherit like POSIX credentials:** a spawn that names no set gets its parent's (`Kernel::child_caps(ppid)`), and every host spawn path attributes the true initiating pid (client `spawn`, `sys.exec`/`npm run` through the shell driver, `child_process` streaming and sync-capture) — so a denied guest cannot shell out to regain `fetch`. The client surface is `os.spawn(argv, { net: false })`.
**Why:** The bit belongs in the kernel even when the mechanism is a worker-side strip — that is exactly the ADR-020 split (kernel decides the numbers/policy, host supplies the mechanism only it has). Enforcement-before-evaluation closes the capture window without a membrane; inheritance closes the laundering-through-children hole, natively testably. And the seam is future-proof: when `Membrane`/`otf:net_fetch` lands, the *same* kernel bit gates it — callers don't change.
**Honest surface (INV-5):** this is **coarse, same-realm, pre-`Membrane`** enforcement — it stops ordinary and careless-malicious code, not a determined realm-surgeon (the `Full` level shares the realm by design, ADR-001/009). It is also all-or-nothing per process: no per-origin allowlists, no audit trail — those arrive only with kernel-mediated egress. Loopback `otf:net_*` (ADR-021) is unaffected: in-instance preview/server traffic is not egress. A denied guest calling `http.request` fails with a TypeError-shaped error, not a faithful `ENETUNREACH` — Node-fidelity refinement tracked in NODE_COMPAT territory.
**Rules out:** treating outbound network as an unconditional ambient right; per-request policy at the `Full` level.
**Native-testable (INV-2):** the grant, the default, and parent-inheritance (including the deny-then-shell-out case) are pure kernel bookkeeping (`cargo test`); the strip, the client knob, and inheritance through the real `exec`/`child_process` paths are proven in headless-browser tests (`tools/net-egress.test.js`).
**Revisit when:** the `Membrane` level lands (move egress behind `otf:net_fetch` with per-origin capability grants + audit), or an embedder needs allowlists rather than a boolean.

### ADR-025 — Process groups are kernel state; the terminal has a foreground pgrp · [locked]
**Context (the fork):** "The foreground pipeline" was a JS `Set` of pids inside the kernel worker — not kernel state at all, despite INV-2 making the process model the kernel's. Consequences: ^C/^Z/SIGWINCH delivery was host bookkeeping that each spawn path had to remember to join (`startProcess` added shell commands; `spawnChild` special-cased `stdio:'inherit'`; anything else was silently outside), an exec'd grandchild's membership depended on which code path created it, `ps` could not show job structure, and the promised `jobs`/`fg`/`bg` (ARCHITECTURE §5) had no substrate. The known symptom: signals and stdin routing around nested interactive processes (`npm create vite` shapes) behaved by accident of the Set's contents.
**Choice:** POSIX process groups, kernel-owned. Every process record carries a **`pgid`**; `Kernel::spawn` takes the `setpgid` shapes — `None` *inherit the parent's group* (the default: children, exec'd sub-commands, `child_process` spawns), `Some(0)` *become a leader* (a new job), `Some(p)` *join group `p`* (later pipeline stages). The controlling terminal carries a **foreground pgrp** (`tcsetpgrp`/`tcgetpgrp`; `0` = the shell owns the terminal). The delivery set for a control-key signal is `pgrp_members(tcgetpgrp())` — live members only. The shell driver places each interactive command/pipeline in a fresh group (leader = first stage) and makes it the foreground group; a shell run acting *for* a process (`sys.exec`, `npm run`) creates **no** new groups, exactly like a non-interactive POSIX shell, so the whole tree stays inside the caller's job. The host's `foreground` Set is gone; `handleExit` returns the terminal to the shell (cooked termios + `tcsetpgrp 0`) when the *last* live member of the foreground group reaps. `ps` reports `pgid`.
**Why:** Signals-to-a-job is the semantic every interactive Unix program assumes; deriving membership from *process ancestry recorded in the kernel* replaces N host-side special cases with one inheritance rule — which is why ^C now reaches arbitrarily deep exec'd grandchildren without any path-specific code. It is also the substrate `jobs`/`fg`/`bg` and SIGPIPE-to-a-group (ADR-023's revisit note) need. And it restores INV-2: the kernel owns the process model; the host asks it who the foreground is.
**Honest surface (INV-5):** no sessions/controlling-terminal *ownership* model yet (one TTY per instance, ADR-021's single-terminal reality) — `pgid` without `sid`; `setsid` arrives if multiple terminals ever do. Background jobs (`&`) still receive the foreground grouping cosmetically (parity with the old Set; real job control — stopped jobs, `fg`/`bg`, `SIGTTIN` — is Phase-3 "later" work this ADR only lays rails for). A grandchild's *stdin* routing (`writeStdin` to a specific pid vs. the shared TTY queue) is unchanged and still per-pid.
**Rules out:** host-side foreground bookkeeping; per-spawn-path signal special cases.
**Native-testable (INV-2):** group placement (leader/inherit/join), live-member delivery sets (zombies drop out), and tcsetpgrp/tcgetpgrp are pure kernel bookkeeping (`cargo test`); pipeline-wide ^C, `ps` pgid visibility, and grandchild signal delivery are proven in headless-browser tests (`tools/pgroups.test.js`).
**Revisit when:** job control lands (`jobs`/`fg`/`bg`, stopped state, `SIGTTIN`/`SIGTTOU`), or a second terminal forces a session (`sid`) model.

### ADR-026 — Userland grows via off-the-shelf Rust (uutils `uu_*` crates) as one wasm multicall binary · [accepted]
**Context (the fork):** ADR-017's revisit condition arrived: the WASI host exists and `/bin/grep` proves the path. The missing everyday utilities (`touch`, `ln`, `base64`, `od`, `split`, …) could be (a) handwritten JS like the /sbin coreutils, (b) the full uutils release wasm (~100 utils, one prebuilt blob), or (c) a WorkerOS-owned crate depending on exactly the `uu_*` crates we want.
**Choice:** (c) — `crates/wsh-utils`: each utility is one Cargo dependency line (`uu_touch = "0.9.0"`) plus one dispatch arm, built to a single `wasm32-wasip1` multicall binary (`/bin/coreutils`, ~6 MB for the first 26). The registry entry carries a `links` list; boot creates `/bin/<name>` symlinks to the one blob and the binary dispatches on argv[0] basename (`coreutils NAME ARGS…` is the explicit fallback). WASI has no cwd, so the host passes the kernel cwd as `PWD` and the binary chdirs to it at startup (wasi-libc's emulated-cwd then makes relative paths correct).
**Why:** Real GNU behavior (flags, exit codes, edge cases) for one dependency line per utility beats re-deriving it in JS; the curated set keeps the blob a fraction of full uutils and excludes wrong-for-the-platform utils (perm bits, mounts, spawn-needing). One blob + symlinks keeps N names at one copy in the VFS (and one CAS chunk set). The tier split of ADR-017 stands: trivial hot-path adapters stay JS; utilities with real logic come from crates.
**Honest surface (INV-5):** dropped for now — `uu_expr` (onig, a C regex), `uu_tac` (mmap), `uu_test`/`uu_stat`/`uu_du` (Unix metadata: mode/uid/dev), `uu_hashsum` (stale on crates.io; `cksum -a sha256|md5|blake2b` covers digests). The WASI host gained the honest file ops this tier needed: `path_symlink`/`path_link`/`path_readlink`, `path_filestat_get` honoring SYMLINK_FOLLOW (lstat), `*_filestat_set_times` (→ kernel utimes), and `fd_filestat_set_size` (userland read+rewrite, the node/fs.js truncate pattern — no kernel ftruncate yet). No Module cache: each spawn compiles the blob (~fine today; cache when it isn't).
**Rules out:** vendoring the full uutils release blob; handwriting GNU-behavior utilities in JS when a `uu_*` crate compiles for wasip1.
**Proven:** `tools/uutils.test.js` (symlink dispatch, relative-path cwd adoption, base64/sha256 vectors, od/nl/paste/comm, tee/split, pipes) — skipped, like grep's, when the binary isn't built (`npm run build:utils`).
**Revisit when:** the blob's per-spawn compile cost shows up in shell loops (add a compiled-Module cache keyed by content), or `uu_stat`/`uu_du`/`uu_test` grow non-Unix fallbacks upstream, or posixutils-rs's `plib::regex` gets a pure-Rust backend (awk/sed).
